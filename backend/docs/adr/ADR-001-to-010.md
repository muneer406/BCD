# ADR-001: On-Device Privacy-First Architecture

## Status
Accepted

## Context
BCD processes highly sensitive medical-adjacent images (breast photos). The Product Requirements Document and Constitution mandate zero-knowledge privacy — the server must never have access to raw images.

## Decision
Use an **on-device-first architecture** with optional encrypted cloud backup.

- **Inference**: MobileViT-S (5M params, int8 quantized, ONNX Runtime) runs on the device
- **Storage**: Embeddings stored locally in SQLCipher-encrypted database
- **Cloud**: Supabase for auth + encrypted blob storage only (server cannot decrypt)
- **Backup**: User-derived key (Argon2id) encrypts embeddings before upload

## Consequences
- Zero marginal cost per user (no cloud inference)
- Works offline
- Privacy from day one — not retrofitted
- MobileViT is less accurate than DINOv2 (trade-off accepted)
- Older/budget phones may have slower inference

## Alternatives Considered
1. Cloud-only inference — rejected (privacy, cost, offline)
2. Hybrid on-device + cloud — accepted for future research opt-in
3. Traditional server-side PyTorch — rejected (privacy, marginal cost)

# ADR-002: MobileViT-S as On-Device Backbone

## Status
Accepted

## Context
The AI pipeline requires a model that runs on-device on budget Android phones ($150 range) while producing embeddings accurate enough for change detection.

## Decision
Use **MobileViT-S** (Apple, 2023) quantized to int8 via ONNX Runtime.

- 5 million parameters
- ~20ms on mid-range phone, ~100ms on budget phone
- 384-dim embedding output
- Replaces current PyTorch EfficientNetV2-S cloud pipeline

## Consequences
- Significant speed improvement (500ms → ~100ms per image)
- Privacy improvement (on-device)
- Accuracy may be slightly lower than EfficientNetV2-S
- Can be upgraded to MobileViT-v2 or newer when available

## Alternatives Considered
1. DINOv2-ViT-B — rejected (86M params, too large for on-device)
2. ConvNeXt-Tiny — rejected (28M params)
3. EfficientNet-Lite — promising, but MobileViT edge in accuracy/size ratio
4. Keep PyTorch cloud — rejected (privacy, marginal cost)

# ADR-003: Tiered Capture (2-Photo Weekly + 6-Photo Monthly)

## Status
Accepted

## Context
The Validation Report identified capture friction as the #1 retention risk. Users will not consistently complete 6-angle sessions weekly.

## Decision
Implement **tiered capture**:
- **Quick check** (weekly): 2 photos (front + dominant side angle)
- **Full session** (monthly): 6 photos (all angles)
- First session: always 6 photos to establish baseline

## Consequences
- Lower barrier to weekly engagement
- Partial sessions are usable (not wasted)
- More data overall (2 weekly + 6 monthly = 14 photos/month vs 6)
- Comparison logic must handle different numbers of angles
- UX must clearly differentiate quick vs full sessions

## Alternatives Considered
1. Full 6-angle only — rejected (friction kills retention)
2. Single photo weekly — insufficient data for comparison
3. Adaptive schedule — deferred to V2 (needs user behavior data)

# ADR-004: Supabase for Auth and Encrypted Storage

## Status
Accepted

## Context
BCD needs auth (email/password) and optionally encrypted cloud storage for backups. Building auth from scratch is not justified.

## Decision
Use **Supabase**:
- **Auth**: Supabase Auth (email/password)
- **Storage**: Supabase Storage for encrypted backup blobs only
- **Database**: Supabase Postgres for metadata only (no images, no embeddings)
- Server never holds decryption keys

## Consequences
- Rapid implementation
- Well-tested auth flows
- Storage costs are manageable (embeddings only, ~1KB per session)
- Vendor dependency — but data is portable (plain Postgres + blob store)
- Supabase can be self-hosted if needed (Constitution: never lock users in)

## Alternatives Considered
1. Firebase — rejected (vendor lock-in, privacy concerns)
2. Self-hosted auth + S3 — viable but adds operational load
3. Auth0 — rejected (cost at scale)

# ADR-005: Quadrant-Based Change Detection

## Status
Accepted

## Context
The AI Research Report concluded that full-image comparison dilutes localized change signals. Clinicians want to know WHERE a change occurred.

## Decision
Divide each breast image into **6 anatomical quadrants**:
1. Upper-outer (left)
2. Upper-inner (left)
3. Lower-outer (left)
4. Lower-inner (left)
5. Right breast analog (mirrored)
6. Areola (both, as separate region)

Each quadrant gets its own change score. The overall score is a weighted combination.

## Consequences
- More explainable results ("change detected in upper-outer quadrant")
- More sensitive to localized changes
- More complex implementation (6x the comparisons)
- SAM segmentation needed for consistent quadrant definition

## Alternatives Considered
1. Full-image comparison — rejected (dilutes localized signal)
2. Fixed grid (e.g., 4x4 tiles) — rejected (not anatomically meaningful)
3. Learned region detection — deferred to V2

# ADR-006: Cycle-Aware Comparison

## Status
Accepted

## Context
Breast tissue changes visibly throughout the menstrual cycle. Comparing across phases produces false flags. The Clinical Advisory Board flagged this as critical.

## Decision
Track the user's menstrual cycle phase and prefer **same-phase comparisons**:
- When a session is captured, log the phase (self-reported or predicted)
- When comparing, prefer baseline sessions from the same phase
- Fall back to nearest-phase if insufficient same-phase data exists
- Flag only changes beyond cycle-normal variation

## Consequences
- Reduces false flags from cyclical changes
- Requires user to log cycle data (or predict from history)
- Adds complexity to comparison engine
- Users without cycle tracking get standard (less accurate) comparison

## Alternatives Considered
1. Ignore cycle — rejected (high false flag rate)
2. Require cycle tracking — rejected (user autonomy)
3. Use only cycle-tracking users' data — viable but reduces data pool

# ADR-007: SAM Segmentation for Breast Region

## Status
Accepted

## Context
Consistent image cropping is critical for meaningful comparison. Phone photos have varying framing, distance, and angle.

## Decision
Use **FastSAM** (lightweight Segment Anything variant) to automatically segment the breast region:
- Runs on-device at ~50ms on mid-range phones
- Segments skin/breast region from background
- Enables consistent anatomical cropping
- Fall back to MediaPipe pose keypoints + center crop if SAM fails

## Consequences
- More consistent cropping than fixed-position approaches
- Works across body types, distances, framing
- Additional ~50ms processing time
- SAM may fail on unusual poses or clothing — fallback required
- FastSAM needs validation on our target phone range ($150 Android)

## Alternatives Considered
1. MediaPipe pose landmarks only — rejected (requires visible upper body)
2. Fixed center crop — rejected (too inconsistent)
3. Manual crop by user — rejected (friction, inconsistency)

# ADR-008: On-Device Local Database (SQLCipher)

## Status
Accepted

## Context
All comparison data must be stored on-device with encryption. The Constitution mandates zero-knowledge privacy.

## Decision
Use **SQLCipher** (an encrypted SQLite variant):
- All embeddings, metadata, and results stored locally
- AES-256-GCM encryption at rest
- Key derived from user's password via Argon2id
- Database never leaves the device unless user explicitly enables encrypted backup

## Consequences
- Strong encryption at rest
- Full offline operation
- Backup requires key management (user must remember password)
- Database could be large after years of sessions (~1MB per year of weekly sessions)
- Migration path needed if database schema changes

## Alternatives Considered
1. IndexedDB (PWA) — rejected (less secure, limited size)
2. Unencrypted SQLite — rejected (Constitution violation)
3. File-based storage (JSON) — not scalable for queries

# ADR-009: Modular AI Pipeline with Versioned Models

## Status
Accepted

## Context
AI models will improve over time. Users need reproducibility — their historical comparisons should not change when the model changes.

## Decision
**Version every model**: Each embedding is tagged with the model version that produced it.
- Comparisons always use embeddings from the same model version
- When a new model ships, all future sessions use it
- Historical sessions keep their original embeddings
- Users can optionally re-process historical sessions with the new model

## Consequences
- Clean separation between model versions
- No unexpected changes to historical results
- Storage increases slightly (multiple versions of same embedding)
- Re-processing requires user action (respects autonomy)

## Alternatives Considered
1. Always use latest model — rejected (historical results change silently)
2. Reprocess all history on model update — rejected (costly, changes user's perceived history)

# ADR-010: Doctor Sharing via One-Time Links

## Status
Accepted

## Context
Users need to share BCD data with their healthcare provider. The solution must be private, simple, and work without the provider installing any app.

## Decision
**One-time shareable link**:
- User generates a link from the app
- Link opens a clinical summary page in any browser
- Link expires after 7 days or one view (whichever comes first)
- Optionally password-protected
- Server proxies encrypted data — never stores decryption key
- User can revoke active links at any time

## Consequences
- Zero friction for doctors (no app install)
- Privacy maintained (link is temporary, revocable)
- Server acts as encrypted proxy — never sees plaintext data
- Link expiry reduces risk of unauthorized access
- Requires server infrastructure for link generation + proxy

## Alternatives Considered
1. PDF export only — accepted as secondary option, link is primary
2. Direct app-to-app sharing — requires doctor to install app (rejected)
3. Email report — less secure, no revoke capability
